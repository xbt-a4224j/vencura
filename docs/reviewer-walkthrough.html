VenCura — Reviewer Walkthrough

A custodial Ethereum-wallet API + admin. This is the 8-step happy path that exercises the four required operations and the three things the design is really about: key custody, correctness under concurrency, and chain-as-source-of-truth. ~10 minutes, no setup.

▶ Open the app ↗ Swagger / OpenAPI ↗ Sepolia Etherscan ↗
Each step reads the same way — Do what to click · See the signal that confirms it · Under the hood a small diagram of what actually happens. Everything runs against the live deployment; no local setup required.

System at a glance

Chain is the source of truth; Postgres is a derived, cached projection. The only deliberate abstraction seam is Signer (custodial → MPC → non-custodial). No Redis — Postgres is also the concurrency substrate (advisory locks + a unique constraint for idempotency).

The walkthrough

1 Open & sign in

Do

Open the app ↗. A Demo mode banner explains auth is bypassed with one shared password. Pick User (choose the account — password is prefilled — and Log in) or Admin for the operator console.

See

You land in the console. Reload the page — you stay signed in and on the same view.

Proves: custodial auth — a per-user JWT; the server-side admin key is never shown to the client.

2 Create & fund a wallet

Do

Go to the Wallets tab. Your wallet is auto-provisioned and master-funded on first load (or create one).

See

A wallet address (0x…) and a small ETH balance. The private key is never shown — by design.

Proves: key custody — the centerpiece. AES-256-GCM at rest behind the pluggable Signer; key decrypted in memory only at sign time, then dropped.

3 Read the balance

Do

On the wallet card, read ETH AVAILABLE; click Refresh.

See

confirmed (with an as-of block) and available = confirmed − pending.

Proves: getBalance — chain-as-truth. The DB never invents a balance; it caches what the chain reports.

4 Sign a message — off-chain proof of ownership

Do

Wallets → Sign a message (a realistic ownership challenge is pre-filled) → Sign, then Verify.

See

An EIP-191 signature, then ✓ verified — recovered 0x… = this wallet. No gas, no transaction.

Why

This is the primitive behind passwordless “Sign-In With Ethereum” and gasless EIP-712 approvals — a custodial wallet proving ownership / authorizing an action without spending gas. The Sign→Verify round-trip is the proof: the signature recovers to exactly this address.

Proves: signMessage — deterministic EIP-191 signing through the same custody seam.

5 Send ETH to vitalik.eth

Do

Wallets → Send: asset ETH, recipient vitalik.eth, amount 0.00001 → Send.

See

The activity row goes PENDING → CONFIRMED with a clickable Etherscan tx link; the ENS name resolved to 0xd8dA…6045.

Proves: sendTransaction end-to-end — ENS, the locked sign/broadcast critical section, and the pending→confirmed lifecycle (a poller, not the request, finalizes).

6 Fire 5 concurrent sends — the nonce lock

Do

Wallets → expand Concurrency demo (nonce lock) → Fire 5 concurrent sends.

See

A green checklist: unique, consecutive nonces · none blocked or dropped · serialized FIFO — and the verdict “5 concurrent sends, zero contention bugs — the lock held.”

Proves: correctness under concurrency — a first-class requirement. The advisory lock serializes the read-nonce→sign→broadcast→persist section per wallet; idempotency is backstopped by a unique constraint.

7 ERC-20 lifecycle — approve & transferFrom

Do

Token tab shows the deployed VCD token + supply. Distribute some VCD to a holder (pick from the dropdown). Switch to the User view → Approve admin for ≥ that amount. Back in Token → transferFrom → admin.

See

Each action gets an Etherscan link. The on-chain allowance is the gate: a pull within allowance+balance succeeds; an over-pull is rejected with a precise message naming the exact shortfall — before any tx is broadcast.

Proves: ERC-20 support with the on-chain allowance as the authorization gate.

Deploying a fresh token and reset/seed are operator-only (they need the admin key). The token above is already deployed, so distribute → approve → transferFrom needs no special access.

8 Activity & audit trail

Do

Open the Activity tab (and the per-wallet feed). Use the Refresh buttons — no full reload needed.

See

One newest-first stream: sends (contract calls labelled by method, e.g. approve), received (inbound transfers indexed from the chain), and governance events (e.g. token.deployed). Logins live in the separate live system log.

Proves: transaction history (on-chain + off-chain) and a durable audit trail — and that incoming transfers, which the platform didn’t initiate, are still surfaced.
