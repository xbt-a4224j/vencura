Custodial wallet API · TypeScript / NestJS · Ethereum Sepolia · built for concurrency
Thesis System Diagram Components Data Model Custody & Keys Operation Flows Concurrency Balances Security Scale Stack & Deploy

VenCura is a backend API that creates and operates custodial Ethereum wallets on users' behalf. Because the platform holds the keys, the architecture is really a study in key custody, transaction correctness, and security — which is what matters most for a custodial wallet platform. This document is the map: components, data model, the custody design, the per-operation flows, and the concurrency model that keeps transactions correct under load.

1Design thesis

Three principles drive every decision:

  • The chain is the source of truth; the database is a derived, cached projection. Balances are read from chain and cached for UX — never invented in the DB.
  • The private key is the asset. It is generated server-side, encrypted at rest (AES-256-GCM), decrypted only transiently in memory at signing time, and never logged or returned. A pluggable Signer lets us swap custody models (encrypted-key → Shamir split → MPC → non-custodial) behind one interface.
  • Correctness under concurrency is a first-class requirement. Ethereum's sequential nonce means concurrent sends on one wallet can collide; idempotency prevents double-broadcast. We design for this explicitly.

2System diagram

Client → NestJS API (modules + async workers) → Postgres (durable state), Postgres advisory locks + a unique-key constraint + scheduled pollers (no Redis), an env master key for encryption, Ethereum/Infura (reads, broadcast, event subscriptions).

5Custody & key management

The heart of the project. A pluggable Signer interface decouples "how we hold the key" from the rest of the system:

interface Signer {
  getAddress(walletId): Promise<Address>
  signMessage(walletId, msg): Promise<Hex>
  signTransaction(walletId, tx): Promise<Hex>
}
BackendHow the key is heldStatus
EncryptedKeySignerAES-256-GCM: the private key is encrypted with a master key held in env/secret. Decrypted in memory only at sign time, then zeroized.Default (built)
ShamirSigner2-of-2 split: the key is reconstructed transiently from two shares (DB + separate store); the full key is never persisted. Demonstrates the "never store the whole key" idea.Bonus (built)
MpcSigner / non-custodialThreshold signatures (key never assembled) / user-held share.Designed in writeup, not built
Why this design: the same DB schema and API support every custody model behind one interface — exactly the spectrum (encrypted-key → Shamir → MPC → non-custodial) that defines their worldview. The signer is the only place key material is ever in plaintext, and only for microseconds.

6Operation flows

createWallet

1
Generate a fresh keypair server-side (viem/generatePrivateKey).
2
Encrypt the private key with AES-256-GCM using the master key from env.
3
Persist address, encrypted_private_key. Return the address only — the key never leaves the server.

getBalance

1
Return the cached confirmed balance instantly + the derived available balance (confirmed − pending outbound − gas reserve).
2
Stale-while-revalidate: kick off an async refresh from chain; push the corrected value when it lands.

signMessage

1
Authorize (caller owns/has access to the wallet).
2
Signer.signMessage decrypts the key in memory with the master key, signs (EIP-191), zeroizes. Return the signature.

sendTransaction (the interesting one)

1
Idempotency: if this Idempotency-Key was seen, return the stored result — don't broadcast twice.
2
Acquire the per-wallet nonce lock (Postgres advisory lock). This serializes all sends for this wallet.
3
Read the authoritative pending nonce + on-chain balance (never the cache) → build tx (native or ERC-20 transfer) → Signer.signTransaction → broadcast.
4
Persist the tx as pending with its nonce; bump next_nonce; release the lock; return transactionHash.
6
A confirmation-watcher poller later sees the receipt, flips status to confirmed/failed, and refreshes the balance cache.

7Concurrency model (the part we deliberately built)

The core problem: Ethereum nonces are strictly sequential per address. Two concurrent sendTransactions on the same wallet that both read nonce N will collide — one tx gets stuck or replaced. And a client retry can broadcast the same payment twice.
  • Per-wallet nonce lock (Postgres advisory lock). A short-lived distributed lock keyed by wallet serializes the read-nonce → sign → broadcast → persist critical section, so each send claims a unique, monotonic nonce. (Postgres advisory lock is the zero-extra-infra alternative; a Redis lock is the documented multi-node scale path.)
  • Idempotency keys. Clients send an Idempotency-Key; the first response is stored and replayed on retries, so a network hiccup never double-spends.
  • Async workers (@nestjs/schedule pollers). A confirmation-watcher reconciles pending txs to confirmed/failed; a balance-refresher updates the cache (event-driven via Transfer logs + new blocks, with a periodic full reconcile as a safety net). Keeping these off the request path keeps the API fast.
  • Authoritative checks on write. Sends validate against live chain state (nonce + balance), never the cache — the cache is for display only.
Right-sized, not over-built: this is the minimum needed to be correct under concurrent sends — a real requirement, not premature scaling. The further scale path (durable-nonce pools, multiple co-signers, provider webhooks) is documented in §10 but intentionally not built.

8Balance model

Three numbers, like a bank account:

  • Confirmed — last value read from chain (cached, with as_of_block).
  • Pending — outbound txs broadcast but not yet mined (and detected incoming).
  • Available = confirmed − pending outbound − gas reserve → what the UI shows as spendable.

On send, we apply an optimistic pending debit immediately for snappy UX, then reconcile to the confirmed value when the watcher sees the receipt. Reads are stale-while-revalidate. Reorgs: a balance isn't treated as final until N confirmations.

10Security considerations

  • Keys: AES-256-GCM encryption at rest; decrypt in memory only; never logged, never returned by the API.
  • Auth & authz: JWT; every wallet action checks ownership/shared-access role.
  • Transport & secrets: TLS; the master encryption key + DB creds in env/secrets manager, never in code.
  • Abuse: rate limiting; idempotency; audit log of every sensitive action.
  • Chain correctness: authoritative nonce/balance checks; reorg-aware confirmations; amounts as bigints.
  • Honest weaknesses (for the writeup): a fully custodial model is a honeypot — a compromised master key + DB is catastrophic; the real mitigation is MPC/non-custodial, which is why the Signer abstraction matters.

11How this scales (documented, not built)

Show the scale path without over-building this:

  • Nonce throughput: per-wallet durable-nonce pools and pre-signed/preprocessed material to pipeline sends.
  • Signing: multiple parallel co-signers; move to MPC/threshold signatures so keys are never assembled (the MPC/threshold-signature model).
  • Refresh: provider webhooks (Alchemy Address Activity) instead of polling; horizontally scaled workers.
  • Multi-chain: a chain-adapter interface (EVM today; Solana/others behind the same Signer + tx-builder seam).
